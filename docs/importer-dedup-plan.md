# План: дедуп дублей ABS в importer

Статус: **план, не реализовано**. Автор-контекст: 2026-06-22, после bulk-заказа Кинга.

## Проблема (наблюдаемая)

При импорте крупных многофайловых аудиокниг (от ~60 файлов: «11 22 63» 320, «Талисман» 152, «Роза Марена» 128, «Стрелок» 66) Audiobookshelf заводит **две library-item записи на одну и ту же папку** (`relPath = "Автор/Название"`):

- одна **каноническая** - с нашей метадатой после `_abs_fix_metadata` PATCH (`author = "Стивен Кинг"`, чистый title);
- одна **сырая** - из встроенных ID3-тегов (в авторы попадают чтецы: «Воротилин Олег, Елисеева Ольга»; title вида «Кинг Стивен. Стрелок»), и она помечается `isMissing`.

Итог для пользователя: книга задваивается в библиотеке + горит вью «Проблемы». Данные при этом целы (обе записи указывают на один набор файлов, хардлинки на месте).

За один вечер дефект проявился дважды (4 книги в bulk-партии + позже «Стрелок» по мере докачки хвоста), оба раза чинился руками: ре-скан снимал ложный `isMissing`, затем `DELETE /api/items/{id}` (db-only, без `hard`) сырого дубля. Воспроизводимо на больших книгах → **systemic, нужен фикс в коде, а не ручная чистка.**

## Гипотеза причины

`cp -al` дерева из сотен файлов не атомарен и занимает заметное время. Собственный folder-watcher ABS успевает среагировать на частично залитую папку и создать первый item из сырых ID3 раньше, чем отрабатывает наш явный `_abs_scan` + `_abs_fix_metadata`. Дальше две записи живут параллельно; PATCH правит только ту, что нашёл `_abs_fix_metadata` первой. Маленькие книги доливаются быстрее, чем срабатывает watcher, поэтому у них дублей нет.

Гипотезу под фикс **точно подтверждать не обязательно**: целевой фикс самовосстанавливающийся и работает независимо от того, почему ABS создал второй item.

## Текущий код (привязка)

`importer.py`:
- `import_torrent()` - hardlink → `_abs_scan` по каждой либе → `_abs_fix_metadata(lib_ids, author, title)` → тег `abk_done`.
- `_abs_fix_metadata()` - поллит recent-25 (`sort=addedAt desc`), находит **первый** item по `relPath == "author/title"`, PATCHит `{metadata:{title, authors:[{name:author}]}}`, возвращает на первом совпадении.

Дедупа нет; на >1 item с одним relPath код реагирует как на один.

## Целевой фикс (основной, self-heal)

Расширить проход правки метадаты до **find-all → выбрать выжившего → PATCH выжившего → DELETE остальных (db-only)**:

1. Собрать **все** items с `relPath == "author/title"` (не первый, а полный список; recent-25 хватает - дубли создаются в одном скане).
2. Выбрать выжившего: приоритет `isMissing == False`; при равенстве - любой детерминированно (напр. меньший `addedAt` или лексикографически меньший `id`).
3. PATCH выжившему - наша каноническая метадата (как сейчас).
4. Для каждого **остального** item с тем же relPath: `DELETE /api/items/{id}` **без `hard=1`** (только запись БД).

### Критично

- **DELETE строго db-only (без `hard`).** Все дубли указывают на ОДИН набор файлов; `hard=1` снёс бы файлы и убил выжившего. Проверено вручную: db-only оставляет файлы (`nlink` не меняется) и выжившего нетронутым.
- **Идемпотентность** - сохранить (повторный прогон не падает; importer и так не перезапускает импорт после `abk_done`).
- **Best-effort, без исключений наружу** - сохранить текущий контракт `_abs_fix_metadata` (ошибки логировать, не ронять импорт).

## Альтернатива/дополнение (превентивно, опционально)

Убрать сам триггер гонки - чтобы items создавались только нашими явными сканами:
- отключить folder-watcher у book-библиотеки в ABS (конфиг ABS, не код) - **минус:** глобально на либу, затронет другие импорты; требует проверки, что наш `_abs_scan` остаётся достаточным;
- либо «осадить» папку перед сканом: после `cp -al` дождаться стабильности (размер/кол-во файлов не меняется N секунд) и только потом `_abs_scan`.

Превенция приятнее, но завязана на поведение/конфиг ABS. **Рекомендация: основной фикс (self-heal дедуп) как обязательный; превенцию - только если после деплоя дубли всё ещё иногда создаются и потом схлопываются (косметическое мелькание во вью «Проблемы»).**

## Тесты (tests/test_importer.py)

- `_abs_fix_metadata` при 2 items с одним relPath → ровно 1 PATCH (выжившему) + 1 DELETE; **DELETE без параметра `hard`** (проверять отсутствие hard в url/params).
- Выживший = не-`isMissing`, если такой есть.
- 1 item (обычная книга) → PATCH есть, DELETE нет (регрессия: не удаляем единственную запись).
- 0 items (скан не успел) → текущее поведение (ретраи, потом лог «not found»), DELETE нет.
- Регрессионный тест на исходный баг: 2 items (один сырой+missing, один канон) → остаётся канон, сырой удалён.

## Критерий готовности

- На искусственной паре дублей в тесте остаётся одна запись с канонической метадатой, файлы не трогаются.
- После деплоя на медиа-хост (`book-importer.service`) следующая крупная книга импортируется без задвоения и без записи во вью «Проблемы».
